# NLVM — Plan d'implémentation

Implémentation du langage NL (spécifié dans [`nlvm-specs`](https://github.com/tivins/nlvm-specs)) : compilateur `nlc`, VM `nlvm`, stdlib native `system.*`, et test runner.

Décisions actées :
- **Langage d'implémentation : Rust** (workspace Cargo multi-crates).
- **Stratégie : tranche verticale** — faire passer un programme minimal de bout en bout (lexer → parser → bytecode → VM → exit code) en premier, avec le test runner YAML comme pilote dès le début, puis élargir milestone par milestone.

Référence roadmap officielle : `nlvm-specs/docs/milestones.md` (9 phases, lexer → optimisations).

---

## Architecture — workspace Cargo

```
nlvm/
├── Cargo.toml            # workspace
├── crates/
│   ├── nl-syntax/        # lexer + parser + AST (M1)
│   ├── nl-sema/          # résolution de noms, typage, les 49 checks (M2)
│   ├── nl-bytecode/      # format .nlm : encodage ET décodage partagés (M3 + loader VM)
│   ├── nl-codegen/       # AST validé → bytecode (M3)
│   ├── nl-vm/            # interpréteur, modèle objet, exceptions, GC, natives (M4–M7)
│   ├── nlc/              # binaire CLI compilateur
│   ├── nlvm/             # binaire CLI VM
│   └── nl-test-runner/   # runner YAML (M8) — binaire `nltest`
└── tests/                # tests YAML supplémentaires (mêmes conventions que les specs)
```

Point clé : `nl-bytecode` est partagé entre compilateur et VM — une seule définition du format `.nlm`, des opcodes et des descripteurs de types, donc pas de dérive entre les deux moitiés. Le runner vit dans le même workspace et lit directement `nlvm-specs/tests/` (chemin configurable).

## Déroulé en tranche verticale

### Phase 0 — Bootstrap
Workspace Cargo, CI (`cargo test` + runner YAML), CLAUDE.md pointant vers les specs.

### Phase 1 — Tranche minimale de bout en bout ⭐ (phase critique)
Objectif : faire passer `m1_0010_parse_minimal` et `m4_0010_minimal_main` (un `main` qui retourne un exit code). Force à poser *tous* les squelettes : lexer/parser minimal, un semblant de sema, émission d'un module `.nlm` valide (magic, constant pool, SHA-256), loader + boucle d'interprétation avec frames, et surtout le **test runner v0** (front matter YAML, blocs `#NLFILE`, comparaison exit code/stdout). À partir de là, chaque avancée se mesure en tests qui passent.

### Phase 2 — VM primitive + expressions (M4)
Arithmétique int/float, comparaisons, contrôle de flux, strings et concaténation, `INVOKE_STATIC`. Cible : `m4_0020` et une batterie de tests maison.

### Phase 3 — Sémantique noyau (M2 partiel)
Typage, résolution de noms, null safety (E003/E004), definite assignment, smart casts — les checks nécessaires aux programmes qu'on sait déjà exécuter. Cible : les tests `m2_*` existants.

### Phase 4 — Objets, tableaux, dispatch (M5)
Classes, héritage, vtables, constructeurs/destructeurs, enums, tableaux. Cible : `m5_*`.

### Phase 5 — Exceptions + closures (M6), puis génériques
Monomorphisation, transverse M2/M3.

### Phase 6 — Stdlib (M7)
Par ordre d'utilité pour les tests : `system.Out/In/Err`, `String`, parsing `Int/Float/Bool`, `List/Map`, puis fichiers, et enfin réseau/threads/process. Cible : `m7_*`.

### Phase 7 — Complétude sémantique
Boucler les 49 codes d'erreur avec **un test YAML par code** (positif + négatif, via `expected_compile_error`) — les specs n'en fournissent que ~15 au total, il faudra en écrire beaucoup.

### Phase 8 — Optimisations (M9, optionnel)
Une fois tout vert.

## Points de vigilance

- La monomorphisation des templates et le `readonly` runtime touchent à la fois le compilateur et la VM — à concevoir dans `nl-bytecode` dès qu'on y arrive, pas après coup.
- ~~Le contrat GC impose des appels de destructeurs : un simple comptage de références ou mark-and-sweep naïf suffira au début, mais il faut choisir tôt.~~ Tranché (2026-07-17) : **comptage de références** — le refcount d'`Arc` est le GC (voir « Destructeurs appelés par le GC » ci-dessous). Limitation assumée : les cycles d'objets ne sont jamais collectés, donc leurs destructeurs ne tournent pas — un collecteur de cycles serait un ajout ultérieur (Phase 8).
- Les tests `m7_0030` (use-after-close) montrent que les specs incluent des exigences de sûreté runtime, pas seulement compile-time.

## Écarts spec/implémentation connus (à traiter)

Repérés en écrivant des démos externes (`nlvm-demos`), donc côté usage réel plutôt que tests internes. Ce sont des trous d'implémentation, pas des ambiguïtés de spec — la spec est claire dans les deux cas.

- ~~**Alias d'import (`use x.Y as Z;`)**~~ — traité (2026-07-17) : `parse_source_file` consomme désormais un `as <ident>` optionnel après le chemin pointé (`nl-syntax/src/parser.rs`) ; `SourceFile.uses` est passé de `Vec<String>` à `Vec<UseDecl { path, alias }>` (`nl-syntax/src/ast.rs`) ; les quatre `import_map`/équivalents (`nl-sema/src/class_table.rs`, `nl-codegen/src/class_table.rs`, `nl-syntax/src/monomorphize.rs`) lient désormais l'alias (quand présent) plutôt que le dernier segment du chemin, et `check_duplicate_imports`/E043 (`nl-sema/src/lib.rs`) compare par nom lié (alias ou simple) donc détecte aussi les collisions entre deux imports aliasés vers le même nom. 2 nouveaux tests (`tests/phase7_0230_import_alias.yaml`, `tests/phase7_0240_import_alias_collision_e043.yaml`, 106/106 avec le reste).
- ~~**Appel de méthode statique sur une classe utilisateur (`Foo.method()`, `Foo` non-`system.*`)**~~ — traité (2026-07-17) : `nl_codegen::class_table::MethodInfo` gagne `is_static` (renseigné depuis `MethodDecl.is_static`, `false` pour les méthodes d'interface). `compile_method_call` (`nl-codegen/src/expr.rs`) reconnaît désormais, juste après le chemin dédié `system.*` existant, le cas où `dotted_path(target)` résout — via `resolve_class_name`/`self.imports` (même résolution que `new`/`catch`, donc gère aussi bien un nom simple same-namespace qu'un alias `use ... as ...`) — à une classe connue de `self.classes` portant une méthode statique de ce nom/arité ; il émet alors un `INVOKE_STATIC` construit à la volée (nouvelle `compile_static_user_call`, calquée sur `compile_super_method_call` mais sans charger de receveur) au lieu de tomber dans `compile_expr(target)` → `resolve_ident` → `unsupported construct: undefined variable`. `nl-sema` n'a nécessité aucun changement : `Expr::MethodCall` sur un `Type::Named` inconnu (le cas `Foo` non résolu comme variable) retombe déjà sur `Type::Void`, traité comme joker par `check_assignable` (pas de faux E004/E009) — la vérification de types/exceptions pour ce chemin reste donc non faite côté sema (déferée à la même lacune que le reste du "unresolved, deferred to nl-codegen" documenté dans `checker.rs`), seul le blocage codegen était réellement en cause. 1 nouveau test bout-en-bout (`tests/phase7_0250_static_method_call_user_class.yaml` : appel statique cross-fichier same-namespace et via alias d'import, 107/107 avec le reste).
- ~~**`readonly` runtime enforcement**~~ — traité (2026-07-17) : `vm.md` § Field access exige que `SET_FIELD` rejette à l'exécution toute écriture sur un champ `readonly` hors constructeur, comme filet de sécurité (« as a safety net; the compiler should have caught this »). Deux trous, pas un seul : `SET_FIELD` (`nl-vm/src/interpreter.rs`) ne consultait jamais le flag `readonly` du champ, *et* `nl-codegen` ne propageait jamais `ClassDecl.is_readonly` (`class readonly X`) dans `Module.class_flags` (toujours `0` pour une classe, cf. `nl-codegen/src/lib.rs`) — donc même en ajoutant le check côté VM, une classe readonly dont aucun champ n'est marqué `readonly` individuellement (cas de la hiérarchie `Exception` du prélude, cf. Phase 7 § E013/E014) serait restée sans protection. Fix : `class_flags: if class.is_readonly { class_flags::READONLY } else { 0 }` en plus du `field_flags::READONLY` par champ déjà câblé ; nouveau `resolve_field_owner` dans `nl-vm/src/interpreter.rs` (même schéma que `resolve_virtual` pour les méthodes) qui remonte la chaîne `extends` depuis la classe *runtime* du receveur jusqu'à trouver la déclaration du champ, pour lire le bon `FieldDescriptor.flags`/`Module.class_flags`. `SET_FIELD` rejette (`VmError::Malformed`, comme les autres invariants "should never happen given a valid compiler" plutôt qu'une exception NL catchable — cohérent avec la formulation vm.md) sauf si l'écriture a lieu dans le `<construct>` de la classe *déclarante* elle-même (`method_flags::CONSTRUCTOR` + FQCN du module courant == FQCN du module propriétaire du champ) sur le receveur exact `this` (`Arc::ptr_eq` avec `locals[0]`) — réplique la règle E013/E014 de `nl-sema` (`checker.rs::check_readonly`) en plus grossier, volontairement, puisque c'est un filet de sécurité et non le check primaire. Comme `nl-sema` bloque déjà toute écriture readonly illégale à la compilation, ce chemin n'est exerçable qu'en contournant `nl_sema::check_compile` — 2 nouveaux tests d'intégration Rust dans `nl-vm/tests/readonly_runtime.rs` (`nl-syntax::parse_source_file` → `nl_codegen::compile_program` directement, sans passer par `nl_sema`, pour simuler le contournement) : un qui vérifie le rejet runtime, un qui vérifie que l'écriture légitime dans le constructeur déclarant continue de fonctionner. 107/107 (tests maison) + 11/14 (nlvm-specs, inchangé) après le changement.
- ~~**Destructeurs appelés par le GC**~~ — traité (2026-07-17). Décision GC actée du même coup (cf. « Points de vigilance ») : **comptage de références**, c'est-à-dire que le refcount des `Arc<Mutex<Object>>` existants *est* le GC — la mémoire était déjà libérée à la chute de la dernière référence, il ne manquait que le crochet `<destruct>`. Implémentation : `impl Drop for Object` (`nl-vm/src/value.rs`) — quand le dernier `Arc` tombe, résolution de `<destruct>` (descripteur `() -> void`) via `resolve_virtual` (passé `pub(crate)`), donc en remontant la chaîne `extends` : un destructeur hérité tourne pour les sous-classes, seul le plus dérivé tourne (la spec ne définit pas de chaînage à la C++). `Drop` n'ayant que `&mut self` (l'`Arc` est déjà mort, impossible de fabriquer un `this`), les champs sont déplacés dans un objet « résurrection » marqué `destroyed: true` qui sert de receveur — le flag garantit « at most once » même si le destructeur re-fait échapper `this` dans une structure vivante. Une exception jetée par le destructeur est silencieusement jetée (`let _ = call_instance(...)`, conforme vm.md). `Object` gagne `program: Weak<Program>` (renseigné seulement par `NEW` dans l'interpréteur — seuls les objets de classes utilisateur ont un `<destruct>` bytecode ; `Weak` et pas `Arc` car les closures de `Thread` capturent `Arc<Program>` *et* des objets, ce qui ferait un cycle) ; les ~20 sites de construction natifs (`native.rs`/`text.rs`/`net_http.rs`/`throw_native`) passent par un nouveau `Object::native(...)` (hook mort, jamais de lookup). Deux corrections d'ordre de verrous en découlent : `SET_FIELD`/`ARRAY_STORE` relâchent désormais le verrou de l'objet/du tableau *avant* de dropper la valeur remplacée (dont le destructeur pourrait rappeler l'objet encore verrouillé → interblocage, `std::sync::Mutex` non réentrant) — risque résiduel documenté du même ordre dans les mutations de collections de `native.rs`, non audité exhaustivement ; et `run_program` (`program.rs`) consomme/droppe le résultat de `main` (dont un éventuel objet d'exception non rattrapée, qui peut avoir un destructeur) *avant* de capturer stdout/stderr, sinon la sortie de ce destructeur serait perdue. Timing obtenu : « prompt » au sens vm.md (sortie de scope = drop des locals du frame, réassignation, `POP`, fin de `main`) ; les cycles ne sont jamais collectés (limitation refcounting assumée, cf. « Points de vigilance »), et les objets retenus par un thread abandonné au retour de `main` ne sont pas détruits non plus. 3 nouveaux tests (`tests/phase7_0260_destructor_gc.yaml` : promptitude + ordre scope/réassignation/fin de main ; `phase7_0270_destructor_transitive_inherited.yaml` : cascade via champs + destructeur hérité via `extends` ; `phase7_0280_destructor_exception_discarded.yaml` : exception de destructeur ignorée, programme continue, stderr vide). 110/110 (tests maison) + 11/14 (nlvm-specs, inchangé).
